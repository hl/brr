$ how the brr workflow works

Agent stages and command gates, chained into one resumable pipeline

A workflow is a versioned YAML pipeline that runs a list of named stages in order. Agent stages run the brr loop engine with a prompt; command stages run a deterministic check — tests, a linter, a build. The agent proposes, the gates validate, and a cycle-back loop sends the pipeline to an earlier stage when later work finds more to do. Progress is saved after every stage, so an interrupted run resumes exactly where it stopped.

The workflow itself is deliberately dumb: it sequences stages, watches for signal files, and records state. All the judgment lives in the prompts the agent stages run.

Anatomy

Workflow files are YAML, resolved .brr/workflows/<name>.yaml first, then the user config dir. Only version: 2 is accepted — older files are rejected with a migration error.

defaults set the fallback profile and iteration cap for agent stages. cycle is optional — omit it for a straight line. stages run top to bottom.

Profile resolves: stage → --profile → defaults.profile → config default. A profile maps to a command + args in .brr.yaml. 

version: 2
description: "requirements -> verified, reviewed code"

defaults:
  profile: claude     # fallback profile for agent stages
  max: 3              # fallback iteration cap

cycle:
  target: build       # jump here on .brr-cycle
  max: 3              # bound the loop

stages:
  - id: spec          # unique id, no / \ ..
    type: agent       # agent | command
    prompt: spec       # file > named prompt > inline
    max: 3            # per-stage cap (overrides default)

  - id: check
    type: command
    command: ["make", "check"]   # argv, no shell

  - id: review
    type: agent
    prompt: review
    max: 1

$ two kinds of stage

agent stage

Runs the brr loop engine: the resolved prompt is piped to the profile's command once per iteration, each iteration in a fresh process with a clean context. It loops until a signal file appears, max iterations is reached, or three consecutive failures occur.

Requires prompt; may set profile and max. Runs with SkipLock — the workflow already holds the exclusive lock.

command stage

Runs an argv array directly — no shell, no expansion — inheriting stdin, stdout, and stderr. These are the deterministic gates: make check, a test runner, a linter.

Requires command; rejects prompt, profile, and max. A non-zero exit stops the whole workflow.

$ how a stage's outcome steers the run

Stage outcomeWhat the workflow doesExit
completes / .brr-complete Advance to the next stage —
.brr-cycle, cycles remain Increment cycle_count, jump to cycle.target —
.brr-cycle, cycle.max reached Log cycle_skipped, warn, advance past this stage —
.brr-cycle, no cycle configured Stop with an error 1
.brr-failed Stop, preserve state, print contents (≤ 4 KiB) 0
.brr-needs-approval Pause, preserve state, print contents (≤ 4 KiB) 0
command stage non-zero exit Stop, preserve state 1
interrupt (Ctrl-C) Stop, preserve state 130

Signal files (.brr-complete, .brr-failed, .brr-needs-approval, .brr-cycle) are detected and removed automatically after each stage.

$ lifecycle & resumable state

Per-stage loop

  1. Resolve prompt + profile (agent) or read argv (command).
  2. Run the stage to completion.
  3. Write <name>.json and append one line to <name>.events.jsonl.
  4. Decide what's next from how the stage ended.

The exclusive lock is acquired once before the first stage and held until the workflow exits.

State under .brr/state/workflows/

<name>.json holds run_id, start_sha, next_stage_id, cycle_count, and per-stage status. <name>.events.jsonl is an append-only event log.

Resume: a valid state file resumes from next_stage_id; invalid state is ignored and the run starts fresh.

Follow-up runs are frictionless: success deletes both files, so the next run starts clean with no --reset. Pause, failure, error, or interrupt preserves both for resume and inspection.

$ brr workflow --help

CommandWhat it does
workflow run <name> Run (or resume) the workflow. Flags: --profile, --notify, --reset.
workflow validate <name> Check schema, profiles, prompts, and command executables — without running anything.
workflow status [name] Print saved state as a stage-by-stage view. --watch redraws live.
workflow init <name> Create .brr/workflows/<name>.yaml from a template (--template ship).
workflow reset <name> Delete the saved state file and event log. No-op if none exist. 
brr workflow init ship          # write .brr/workflows/ship.yaml
brr workflow validate ship      # schema, profiles, prompts, commands
brr workflow run ship           # run the pipeline, live flow shown
brr workflow status ship --watch # follow a long run live
